## 第一部分：项目整体介绍（30秒版）

### Q：请简单介绍一下你做的这个项目。

**A：**

我做的项目叫 **AIStu——校园AI教务助手平台**，是一个面向中小学的 **一站式教务管理与AI智能辅助平台**。

简单来说，就是把教务系统里的学生管理、教师管理、班级管理、成绩管理这些常规功能，跟 **AI能力** 结合起来。平台里集成了 **八个AI工具**，比如自动写期末评语、生成纪律辅导话术、润色通知、成绩波动诊断、模拟面试这些，覆盖老师日常的文本工作。

但整个项目里 **最核心、最拿得出手的模块，是「学生关怀」模块**。

这个模块解决一个真实痛点：很多学生出问题不是一天突然爆发的，而是从 **安全事件 → 情绪受损 → 社交退缩 → 学习影响 → 行为波动**，一条链慢慢传导出来的。传统系统只看单维度异常，老师很难提前发现。我们这个模块就是 **把零散线索拼成完整画像，并且能预判风险怎么传导**。

> （补充：技术栈是 FastAPI + MySQL + LangGraph + Neo4j + Vue3，前后端全栈）

## 第二部分：学生关怀模块精讲（2分钟版）

### Q：这个学生关怀模块具体怎么工作的？

**A：**

我们的核心思路是 **分层次处理**，不是一个大黑盒直接出结果。一共分四个层次：

**第一层：规则层——做基础画像**

我们把风险拆成六个维度：**情绪状态、社交融入、校园安全、家庭支持、学习压力、行为稳定**。系统根据出勤记录、行为事件、成绩波动、家校沟通这些数据，给每个维度打分，合成一个基础画像。这部分是 **确定性规则**，可解释、可追溯。

> （补充：比如迟到扣多少分、打架扣多少分，都是写死的规则，老师能看懂）

**第二层：贝叶斯修正层——做概率推断**

规则层有个硬伤——它只能看到已经发生的问题。但老师真正需要的是 **提前知道谁正在变危险**。

所以我们引入了 **贝叶斯推断**。简单说就是：有了新证据之后，更新对这个学生风险的判断。比如学生在AI对话里说了句"我被打了"，系统就会自动把这个证据转化成一个 **似然比**，去更新安全维度的后验概率。

这一层覆盖四个高敏感维度：**情绪、安全、家庭、社交**。

> （补充：我们用了边际递减，避免同类证据反复累加；用了时间衰减，超过90天的事件权重降到20%以下）

**第三层：恶性事件传导层——做跨维度扩散**

这是最大亮点。当一个维度检测到恶性事件时，比如校园安全有欺凌，系统不会只看安全维度，还会自动判断这件事 **会不会传导到情绪、社交、学习、行为**。

**恶性事件传导流程图：**

```
              恶性事件命中（欺凌/威胁/高等级冲突）
                           ↓
               ┌───────────┼───────────┐
               ↓           ↓           ↓
         safety=0.45   AI摘要      图谱冲突
               ↓           ↓           ↓
        ┌──────┴──────────┴──────────┐
        ↓        ↓        ↓         ↓
   emotion   social    study    behavior
   溢出0.30  退缩0.25  影响0.16  波动0.18
        ↓        ↓        ↓         ↓
   原始风险 vs 次生风险 → 分列展示给老师
```

这样老师看到的不只是一个分，而是 **原始风险和次生风险分列展示**，知道哪些是根因，哪些是被带出来的。

> （补充：还有保护性证据机制，老师复核确认「已解决」或「误报」之后，系统会降低风险，不会只会干报警）

**第四层：多智能体解释层——给老师可理解的结论**

最后我们用了 **LangGraph**，构建了一个 **6专家并行 + 1整合** 的状态图。六个专家节点分别对应六个维度，**并行调用大模型**，各自分析自己维度的证据，然后汇聚到整合节点，生成一段老师能直接理解的结论。

**LangGraph并行流程图：**

```
                        START
          ┌──────────────┼──────────────┐
          ↓              ↓              ↓
   emotion_agent   social_agent   safety_agent
          ↓              ↓              ↓
   family_agent    study_agent  behavior_agent
          └──────────────┼──────────────┘
                         ↓
                integration_agent
                         ↓
                        END
          （6个并行 → 自动汇聚 → 整合输出）
```

> （补充：总共耗时大约3秒，比串行6次快了5倍）

## 第三部分：面试常考知识点拆解

### Q1：你们这个贝叶斯推断具体是怎么实现的？

**A：**

我们用的是 **赔率形式的贝叶斯更新**，公式很简单：

**后验赔率 = 先验赔率 × 似然比**

**计算流程图：**

```
先验概率 P(R)          证据列表 E₁,E₂,...Eₙ
     ↓                       ↓
转赔率 O=P/(1-P)       匹配似然比 LR₁,LR₂,...
     ↓                       ↓
     └─────── 累乘 ──────────┘
                  ↓
          后验赔率 = O × LR₁ × LR₂ × ...
                  ↓
          转概率 P = O'/(1+O')
                  ↓
           最终后验概率
```

其中赔率 Odds = P / (1-P)，似然比 LR = P(证据|有风险) / P(证据|无风险)。

具体流程是：
1. **取先验概率**：系统默认配置，比如安全维度的先验是 0.15
2. **转成赔率**：0.15 / 0.85 ≈ 0.176
3. **匹配证据**：从所有信号里找这个维度的证据，每条证据有自己的 LR 值
4. **累乘 LR**：后验赔率 = 先验赔率 × LR₁ × LR₂ × ...
5. **转回概率**：P = Odds / (1 + Odds)

举个例子：如果安全维度先验 0.15，命中欺凌事件 LR=3.8，后验就是：
- 赔率：0.176 × 3.8 ≈ 0.67
- 概率：0.67 / 1.67 ≈ 0.40

概率从 15% 涨到 40%，系统就认为这个人安全风险明显上升了。

> 追问：那你怎么处理证据相关性？
> 答：用了边际递减。n条同类证据，每条LR开 1/√n 次方，比如3条同类证据，每条LR^(1/√3) ≈ LR^0.577。这样第三条证据的增量效应远小于第一条。

### Q2：为什么不用深度学习做情感分析，而用词典方法？

**A：**

这个问题很好，当时我们也纠结过。最后选词典方法，原因有三：

**第一，可解释性要求高**。教育场景下，老师必须知道系统为什么判定某个学生有问题。词典方法可以直接告诉老师"因为某某关键词"，但深度学习是个黑盒，说不清楚。

**第二，数据够用**。我们的文本来源主要是教师评语、出勤备注、AI对话摘要，这些文本格式规范、用词集中，词典覆盖率足够。不像社交媒体那种需要深度语义理解的场景。

**第三，快速迭代**。学校的需求会变，今天发现某个关键词漏了，加进去就行，不需要重新训练模型。

> 总结：**在可解释性要求高、数据规范、需要快速迭代的教育场景，词典方法比深度学习更实用。**

### Q3：六个专家节点算Agent吗？

**A：**

**说实话，不算真正的Agent，本质上就是LLM调用，是封装成函数调用的Prompt。**

真正的Agent有四大特征：**自主性、反应性、主动性、社交性**。我们的专家节点只做一件事：接收结构化数据，输出结构化JSON，相当于一个函数映射 f: Input → Output。

但这不是偷懒，而是**有意识的设计选择**。因为：
- 任务复杂度低，用Prompt就够了
- 省成本，一次调用2秒搞定
- 输出稳定可控，不会跑偏

如果未来需要，可以在专家节点里注入工具调用能力（比如查数据库、调API），那时才能算真正的Agent。

### Q4：时间衰减和边际递减，两者有什么区别？为什么都要？

**A：**

两者是**正交的，解决不同维度的衰减问题**。

**对比流程图：**

```
           时间衰减                            边际递减
       （解决时间维度）                    （解决数量维度）
                                                
  事件距今越远 → 权重越低            同类证据越多 → 单条增量越小
       ↓                                    ↓
  ┌─────────────┐                    ┌──────────────────┐
  │ 7天内  100% │                    │ 第1条 LR^1.0     │
  │ 8-30天 75%  │                    │ 第2条 LR^0.707   │
  │ 31-90天 45% │                    │ 第3条 LR^0.577   │
  │ >90天  20%  │                    │ 第n条 LR^(1/√n)  │
  └─────────────┘                    └──────────────────┘
       ↓                                    ↓
  处理"旧事件干扰"                    处理"同类事件重复计分"
```

**时间衰减**解决的是 **时间维度**：越旧的事件越不重要。
**边际递减**解决的是 **数量维度**：同类事件重复出现时，效应会递减。比如连续5次迟到，不应该简单累加5倍，因为很可能都是同一个原因导致的。

**为什么两个都要？**
- 如果只有时间衰减：近7天迟到5次，权重 5×0.1×1.0=0.5，高估了（可能是同一原因）
- 如果只有边际递减：半年前打架 + 昨天打架，权重被历史事件拖累，拉低了昨天事件的紧迫性

> 所以一定是 **先做时间衰减，再做边际递减**，两个串联。

```
原始事件 ──→ 时间衰减 ──→ W' ──→ 边际递减 ──→ W'' (最终权重)
              (关心当下)         (避免重复计分)
```

### Q5：教师复核闭环有什么意义？

**A：**

核心是解决 **人机协同和责任归属** 问题。

**闭环流程图：**

```
┌──────────────────────────────────────────────────────┐
│                   教师复核闭环                         │
│                                                      │
│  AI研判结果 ──→ 教师复核 ──→ 四种状态：              │
│                              ├─ confirmed  (确认)    │
│                              ├─ false_alarm (误报)   │
│                              ├─ resolved   (已解决)  │
│                              └─ in_progress (处理中) │
│                                      ↓               │
│                              复核结果反馈             │
│                              ├─ 作为贝叶斯新证据      │
│                              │  (resolved→LR<1)      │
│                              │  (false_alarm→LR<<1)  │
│                              └─ 规则权重优化          │
│                                 (系统越用越准)         │
└──────────────────────────────────────────────────────┘
```

教育决策涉及未成年人，责任很重。如果AI自动判断一个学生高风险，老师直接去干预，万一AI错了怎么办？

所以我们设计的是 **AI推荐 + 教师确认** 模式。这样有两个好处：
1. **责任明确**：最终决定权在老师手上
2. **持续优化**：老师的复核结果可以作为新证据反馈到贝叶斯模型里，系统越用越准

> 补充：教师确认误报（false_alarm）对应 LR=0.15-0.45，已解决（resolved）对应 LR=0.35-0.55，这些保护性证据会显著降低后验概率。

### Q6：用LangGraph对比直接用LangChain有什么优势？

**A：**

最大的优势是 **并行执行**。

**对比流程图：**

```
LangChain（串行链）：                    LangGraph（并行DAG）：
                                            
emotion → social → safety             emotion ──┐
    → family → study                  social  ──┤
    → behavior → 整合                 safety  ──┼→ 整合
                                     family  ──┤
耗时：6 × T ≈ 14秒                    study   ──┤
                                      behavior ──┘
                                     耗时：max(T) + T整合 ≈ 3秒
                                     加速比：~5倍
```

LangChain是串行链条，6个专家节点一个一个跑，总共要 6 × 单个耗时。LangGraph是 **有向无环图（DAG）**，6个专家节点可以同时并行。

另外，LangGraph使用 **状态图（StateGraph）** 的模式，所有并行节点的输出通过 `Annotated[list[dict], operator.add]` 幺半群运算自动合并，类型安全，不容易出错。

### Q7：安全披露捕获是怎么工作的？

**A：**

这是系统中一个独立但非常重要的入口，负责从 **文本侧** 提前捕获高危信号。

**安全披露捕获流程图：**

```
学生AI对话/教师备注/观察记录
          ↓
    ┌─────────────────────┐
    │  关键词检测引擎       │
    │  高危词：被打/打我/   │
    │  受伤/流血/欺负/霸凌/ │
    │  威胁/校园暴力        │
    │                     │
    │  排除误判：           │
    │  打游戏/打卡/打电话   │
    └─────────────────────┘
          ↓ 命中
    自动生成关怀信号
    ├─ safety维度: 权重0.75
    └─ emotion维度: 权重0.4
          ↓
    触发画像重算
```

很多真正需要老师介入的场景，并不会先出现在"违纪表"里，而是先出现在 AI 对话、观察记录这些比较碎的文本线索里。如果系统只看结构化数据，很容易错过。安全披露捕获就是 **把文本里的高危苗头提前送进主链路**。

### Q8：社交孤立贝叶斯网络是怎么设计的？

**A：**

社交孤立是单独做了专项插件，里面挂了一条专门的贝叶斯因果链：

**社交孤立贝叶斯因果图：**

```
          peer_disconnect（同伴连接缺失）
                    ↓ impact=0.92
           ┌────────┴────────┐
           ↓                 ↓
  emotional_withdrawal  family_support_gap
    （情绪退缩）          （家庭支持不足）
        ↓                     ↓
  safety_threat  ←────  behavior_retreat
  （安全威胁暴露）       （行为退避）
              ↓
      SOCIAL_ISOLATION
      （社交孤立判定）
```

这个网络不只是回答"有没有孤立"，而是回答：
- 当前的孤立风险是 **暂时社交退缩** 还是 **持续边缘化**？
- 哪些证据是风险证据，哪些是 **保护性证据**（比如已有同伴支持）？
- 判断有多可靠？（会输出 **覆盖度、缺失项、置信度**）

这样老师拿到的不是一个简单的"孤立/不孤立"二分法，而是一个可以追溯的因果链分析。

## 第四部分：高频追问与应对

### Q9：你这个系统数据量有多大？能支撑多少学生？

**A：**

我们现在是原型阶段，但设计上考虑了扩展性。单个学生的关怀画像数据量很小，主要瓶颈在 **LLM 调用**。如果学校有2000个学生全部跑一遍研判，大概需要 2000×3秒 ≈ 1.7小时。

实际场景下不需要全量跑，我们设计的是 **按需触发**：只有检测到新信号时才重算。一般来说，一个班级每天活跃的学生也就十几个，完全来得及。

### Q10：怎么评估你们这个系统的效果？

**A：**

主要通过 **教师复核一致率** 来评估：

```
评估闭环：
                                    教师确认  → TP（真阳性）
AI判高风险 ──→ 教师复核 ──→
                                    教师否认  → FP（假阳性/误报）

核心指标：
- 精确率 Precision = TP / (TP + FP)
- 一致率 Agreement = (TP + TN) / Total
- 当前内部测试一致率 ≈ 70-80%
```

主要误报来自 **数据缺失**——有些维度没有信号，系统用了默认先验，但实际状态可能完全正常。

优化方向：
1. 补充数据源，降低缺失维度的权重
2. 把教师复核数据反馈回模型，持续调优

### Q11：如果让你继续优化，你会怎么做？

**A：**

四个方向：

**第一，专家分级调用**。不是每次六个专家都跑，而是先快速过一遍规则，判断哪些维度确实有异常，只调异常的专家，能省 30-50% 的LLM调用。

```
规则快速扫描 ──→ 异常维度？──→ 是 ──→ 只调该维度专家
                        └──→ 否 ──→ 跳过，用规则结果
```

**第二，动态模型选择**。高风险场景用强模型，低风险场景用轻量模型，平衡成本和效果。

**第三，增量分析**。不需要每次全量重算，只处理新增信号，减少 token 消耗。

**第四，历史记忆注入**。这次的分析结果可以注入到下次的 prompt 里，让模型理解变化趋势，而不是每次都从头看。
