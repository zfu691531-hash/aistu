# 基于学生画像的分班分组优化方案

> 版本：v1.1  
> 日期：2026-04-10  
> 状态：可实施方案  
> 适用范围：学生关怀模块能力复用 + 分班/分组能力增强

---

## 一、方案结论

本方案总体可行，但不建议一次性上线“画像均衡 + 好友优先 + 冲突规避 + 学习互助 + AI个性化建议”全量能力。  
结合当前系统现状，建议采用 **分阶段落地** 策略：

- **V1 首先落地**：基于现有学生关怀画像，增强分班与分组的风险均衡能力
- **V2 再扩展**：接入社交关系数据，支持冲突规避与关系优化
- **V3 再探索**：引入历史效果反馈和 AI 辅助建议

其中，**V1 是明确可做、可验证、可交付的**；V2 和 V3 应作为后续迭代项，而不是首版上线范围。

---

## 二、现状分析

### 2.1 现有分班分组能力

| 模块 | 文件 | 当前能力 | 当前均衡因素 |
|------|------|----------|-------------|
| 校务分班 | `placement_service.py` | 新生正式分班结果校验、确认、落库 | 性别、平均成绩 |
| 教师分组 | `grouping_service.py` | 班级内分组方案保存、更新、校验 | 性别、平均成绩 |

当前系统中的分班分组服务更偏向 **结果校验与方案保存**，算法能力较轻，尚未形成独立的智能优化层。

### 2.2 现有学生关怀能力

学生关怀模块已具备以下基础：

- 六维学生关怀画像：情绪、社交、安全、家庭、学习、行为
- 风险等级：`low / medium / high / critical`
- 恶性事件识别与风险传导
- 贝叶斯辅助判断与前瞻风险提示

这意味着分班分组优化所需的“画像主数据”已经存在，V1 不需要从零建设画像体系。

### 2.3 当前问题

- 分班/分组仅考虑性别和成绩，维度单一
- 高风险学生可能集中在少数班级或小组
- 现有分组逻辑未利用学生关怀画像
- 社交关系能力在关怀模块中有概念基础，但在分班场景中尚未形成稳定、结构化、可审计的数据来源

---

## 三、建设目标

### 3.1 V1 核心目标

V1 只解决三个问题：

1. **风险分散**：避免高风险学生集中到单个班级或小组
2. **基础均衡**：在性别、成绩、画像风险三个维度上保持整体均衡
3. **结果可解释**：输出分班/分组报告，说明方案为何成立

### 3.2 暂不纳入 V1 的目标

以下目标有价值，但不应纳入首版交付：

- 好友优先同班/同组
- AI 自动推断社交关系并参与分配
- 个性化大模型分班建议
- 学期中自动动态调班
- 基于历史成效的自动调权

这些能力需要更高的数据可靠性和更复杂的产品治理，不适合作为首版上线范围。

## 四、数据基础与字段口径

### 4.1 可直接复用的数据

V1 使用以下已有数据：

| 类别 | 字段/来源 | 用途 |
|------|-----------|------|
| 基础信息 | 学生性别、班级、年级 | 基础均衡 |
| 学业数据 | 平均成绩 | 学业均衡 |
| 画像数据 | `emotion_score` `social_score` `safety_score` `family_score` `study_score` `behavior_score` | 风险分层与均衡 |
| 画像标签 | `risk_level` | 高风险学生分散 |

### 4.2 分值语义约定

必须统一一个前提：

- 学生关怀画像字段本身表示 **风险分**
- 值越高，表示该维度风险越高
- 分班算法中不要把原始字段改写成“越高越健康”的口径

为了避免与现有关怀模块认知冲突，建议在分班逻辑中显式引入两个概念：

- `risk_score`：直接使用或加权汇总现有画像风险分，值越高风险越高
- `support_score`：基于低风险程度和学业能力构造的支持分，值越高越适合作为稳定搭配对象

### 4.3 缺失数据处理原则

对于缺失画像数据，V1 不建议直接使用默认值参与关键决策。建议采用以下策略：

- 缺失维度不参与该维度优化
- 学生仍可参与分班，但在结果报告中标记“画像不完整”
- 如果高风险等级字段缺失，则不进入高风险分散逻辑
- 后台向管理员提示数据补齐，而不是静默填默认值

这样更稳，也更符合教育场景下的可解释要求。

## 五、V1 算法设计

### 5.1 设计原则

V1 不追求全局最优，只追求：

- 结果稳定
- 逻辑可解释
- 复杂度可控
- 可人工复核

因此建议采用 **规则分层 + 蛇形分配 + 局部交换优化** 的组合方案，而不是直接引入复杂优化算法。

### 5.2 学生分层

先对学生进行分层：

1. `critical`：必须优先分散
2. `high`：优先分散
3. `medium`：普通均衡
4. `low`：普通均衡

分层优先依据：

- 第一优先：`risk_level`
- 第二优先：综合风险分 `risk_score`
- 第三优先：成绩分层

### 5.3 两类评分

#### 1. 风险分

```python
def calculate_risk_score(profile):
    return (
        0.20 * profile.emotion_score +
        0.20 * profile.social_score +
        0.20 * profile.safety_score +
        0.15 * profile.family_score +
        0.15 * profile.study_score +
        0.10 * profile.behavior_score
    )
```

说明：

- 直接延续学生关怀模块“越高越风险”的字段语义
- 权重先采用业务规则设定，后续可调
- V1 不做自动学习调参

#### 2. 支持分

```python
def calculate_support_score(student, profile, score_max=100):
    academic_support = (student.avg_score or 0) / score_max
    low_risk_support = 1 - calculate_risk_score(profile)
    return 0.6 * low_risk_support + 0.4 * academic_support
```

说明：

- 支持分用于组内搭配，不用于替代风险分
- 支持分高的学生，更适合与高风险或学困学生同组

### 5.4 分班主流程

#### 步骤 1：基础准备

- 获取待分班学生列表
- 拉取学生画像与平均成绩
- 标记画像缺失学生
- 计算每名学生的 `risk_score` 与 `support_score`

#### 步骤 2：高风险分散

- 先处理 `critical`
- 再处理 `high`
- 采用蛇形分配，优先投放到当前高风险人数最少的班级

#### 步骤 3：普通学生填充

- 按成绩分层后蛇形填充
- 填充过程中同步检查：
  - 班级人数
  - 性别比例
  - 班级平均成绩
  - 班级平均风险分

#### 步骤 4：局部交换优化

在初始方案生成后，对班级间做小范围交换，目标是降低以下指标：

- 班级间高风险人数差异
- 班级间平均成绩差异
- 班级间平均风险分差异

### 5.5 均衡性评估口径

V1 的均衡评估应聚焦 **班与班之间的差异**，而不是单个班内部方差。

建议输出以下指标：

- 各班人数差
- 各班男女比例差
- 各班平均成绩差
- 各班高风险人数差
- 各班平均风险分差

可采用如下思路：

```python
def evaluate_plan_balance(class_metrics):
    return (
        - 1.0 * variance(class_metrics["student_count"]) 
        - 1.0 * variance(class_metrics["male_ratio"])
        - 1.2 * variance(class_metrics["avg_score"])
        - 1.5 * variance(class_metrics["high_risk_count"])
        - 1.5 * variance(class_metrics["avg_risk_score"])
    )
```

这里的核心是比较 **各班指标序列的方差**。

### 5.6 分组主流程

班级内分组的 V1 目标比校务分班更聚焦：

- 每组都有基础学业支撑
- 高风险学生不过度集中
- 学困学生尽量与支持分较高的学生同组

流程如下：

1. 先按组数生成空组
2. 优先分配高风险学生，每组尽量均匀
3. 再分配支持分高的学生，补足每组稳定支撑
4. 最后填充分配剩余学生
5. 局部交换，修正极端不均衡情况

## 六、约束设计

### 6.1 V1 硬约束

| 约束 | 说明 |
|------|------|
| 班级/小组人数不超上限 | 必须满足 |
| 分班结果完整覆盖学生 | 必须满足 |
| 同一学生不能重复分配 | 必须满足 |
| 性别比例差异不超过设定阈值 | 默认 10% |

### 6.2 V1 软约束

| 约束 | 说明 |
|------|------|
| 高风险学生尽量均匀分散 | 目标约束 |
| 平均成绩差异尽量缩小 | 目标约束 |
| 平均风险分差异尽量缩小 | 目标约束 |
| 学困学生尽量匹配支持分高者 | 仅分组场景 |

### 6.3 不纳入 V1 的约束

- 好友同班/同组优先
- 冲突学生自动规避
- 基于 AI 推断关系的自动编排

这些约束要依赖高质量社交关系数据，建议放到 V2。

## 七、模块设计

### 7.1 建议新增模块

| 文件 | 功能 |
|------|------|
| `services/profile_placement_service.py` | 基于画像的分班方案生成 |
| `services/profile_grouping_service.py` | 基于画像的班级内分组方案生成 |
| `services/profile_balance_optimizer.py` | 评分、分层、交换优化、报告输出 |

### 7.2 建议能力边界

现有服务职责不变：

- `placement_service.py`：继续负责校验、确认、落库
- `grouping_service.py`：继续负责方案保存、更新、查询

新增服务职责：

- 负责“生成候选方案”
- 输出结构化摘要与均衡报告
- 不直接替代现有确认链路

这样可以避免对现有流程做破坏性改造。

### 7.3 社交关系数据模型建议

`StudentSocialRelation` 可以保留为 V2 设计，但不建议作为 V1 强依赖。

如果后续推进，建议先只支持：

- `conflict`
- `manual_confirmed`
- `source_traceable`

不建议首版就直接启用 `ai_inferred friend/conflict` 参与自动分配。

## 八、API 设计建议

### 8.1 V1 分班接口

```json
POST /api/placement/generate-with-profile
{
  "grade": "2026",
  "target_classes": [1, 2, 3, 4],
  "constraints": {
    "max_gender_ratio_gap": 0.10,
    "balance_academic": true,
    "balance_risk": true,
    "disperse_high_risk": true
  }
}
```

响应中建议新增：

- 分班结果
- 各班摘要
- 画像缺失名单
- 风险均衡报告
- 人工复核提示

### 8.2 V1 分组接口

```json
POST /api/grouping/generate-with-profile
{
  "class_id": 1,
  "group_count": 6,
  "constraints": {
    "balance_academic": true,
    "balance_risk": true,
    "ensure_support_distribution": true
  }
}
```

### 8.3 V2 预留字段

后续可预留但不启用：

- `avoid_conflicts`
- `prefer_friends_together`
- `relation_source`

## 九、实施计划

### 9.1 V1 真实排期建议

相较原方案，建议把排期从“14 天理想值”调整为“2 到 3 周可交付”。

### 9.2 分阶段安排

#### 阶段一：方案生成能力（3-4 天）

| 任务 | 输出 |
|------|------|
| 梳理画像取数与成绩取数逻辑 | 数据装配层 |
| 实现风险分与支持分计算 | 评分工具 |
| 实现分层与蛇形分配 | 初版分班/分组算法 |
| 输出均衡摘要 | 报告结构 |

#### 阶段二：局部优化与接口接入（3-4 天）

| 任务 | 输出 |
|------|------|
| 实现局部交换优化 | 优化逻辑 |
| 接入分班生成接口 | API |
| 接入分组生成接口 | API |
| 接入现有校验与确认流程 | 可闭环调用 |

#### 阶段三：测试与报告（3-5 天）

| 任务 | 输出 |
|------|------|
| 单元测试 | 核心算法测试 |
| 集成测试 | 接口测试 |
| 构造样例数据 | 演示用数据 |
| 生成使用说明与结果示例 | 文档 |

#### 阶段四：试运行与调参（3-5 天）

| 任务 | 输出 |
|------|------|
| 教师/管理员试用 | 反馈记录 |
| 调整权重和阈值 | 参数配置 |
| 修正极端案例 | 稳定版 |

## 十、预期输出

V1 上线后，系统应至少输出以下结果：

### 10.1 分班结果

- 每个班的学生名单
- 每班人数、男女比例、平均成绩
- 每班高风险人数、平均风险分

### 10.2 分组结果

- 每组学生名单
- 每组平均成绩
- 每组平均风险分
- 每组支持分较高学生名单

### 10.3 均衡性报告

- 是否满足硬约束
- 各班主要均衡指标对比
- 是否存在画像缺失影响
- 建议人工复核的班级或小组

## 十一、风险评估

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| 画像数据缺失 | 影响风险分层准确性 | 缺失维度不参与优化，输出缺失报告 |
| 风险权重设置不合理 | 分配结果不符合业务直觉 | 先固定规则权重，试运行后调参 |
| 极端个案无法自动处理 | 自动方案不稳定 | 输出人工复核提示，允许手工调整 |
| 教师对画像参与分班敏感 | 接受度风险 | 强调“辅助生成 + 人工确认” |
| 社交关系数据不可靠 | 错误规避或错误搭配 | V1 不纳入强依赖，V2 再接入 |

---

## 十二、后续扩展路线

### 12.1 V2

- 引入人工确认的冲突关系
- 分班中支持冲突规避
- 分组中支持冲突规避与简单关系优化

### 12.2 V3

- 基于历史分班效果做参数校准
- 基于教师反馈做方案效果回收
- 探索 AI 辅助说明，而不是 AI 直接决定分配

### 12.3 不建议优先做的方向

- 直接用大模型生成最终分班结果
- 在缺乏可审计关系数据前做好友优先
- 在首版引入复杂全局优化算法

---

## 十三、最终建议

这项能力值得做，但应按以下原则推进：

- **先做 V1，可交付优先**
- **先做风险均衡，不做关系黑箱**
- **先做辅助生成，不替代人工确认**
- **先做解释报告，再谈智能推荐**

如果按此版本推进，方案与现有学生关怀模块是衔接顺畅的，且能在较低风险下形成新的产品亮点。
