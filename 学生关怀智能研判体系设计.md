# 学生关怀智能研判体系设计

这份文档用于说明学生关怀模块的整体业务闭环、核心技术结构和主要实现边界。它比 `README.md` 更聚焦学生关怀，又比 `学生关怀关键技术亮点说明.md` 更偏设计和实现。

## 1. 功能定位

学生关怀模块的目标不是直接替代教师判断，而是把分散在校园业务中的风险线索重新组织起来，帮助老师更早发现、解释和跟踪需要关注的学生。

系统当前重点覆盖六个维度：

- 情绪状态
- 社交融入
- 校园安全
- 家庭支持
- 学习压力
- 行为稳定

它最终提供三类结果：

1. 学生画像：汇总学生在六个维度上的基础状态。
2. 智能研判：对风险原因、表现和建议进行结构化解释。
3. 复核评估：记录教师判断并反向评估系统表现。

## 2. 业务闭环

模块围绕“事实进入系统之后如何走到教师动作”来设计，而不是围绕单次推理接口来设计。完整闭环如下：

1. 数据采集：采集学生基础档案、成绩、考勤、行为事件、家校沟通、AI 摘要等信息。
2. 信号抽取：把原始记录转成可用于判断的风险信号和保护性信号。
3. 画像生成：形成六维画像、风险概览和重点观察点。
4. 智能研判：通过多智能体分别解释不同维度，再做综合汇总。
5. 教师复核：教师确认真风险、误报、处理中或已解决状态。
6. 评估优化：统计规则命中、复核一致率和场景分布，为后续调优提供依据。

这条闭环意味着学生关怀不是“算一次分数就结束”，而是一种持续更新的研判机制。

## 3. 数据输入面

系统目前主要消费以下几类数据：

| 数据类型 | 典型来源 | 作用 |
| --- | --- | --- |
| 学生基础信息 | 学生档案、班级信息 | 提供身份与组织上下文 |
| 成绩数据 | 考试成绩、批次对比 | 识别学习波动与持续下滑 |
| 考勤数据 | 迟到、缺勤、请假、早退 | 识别出勤异常与节律变化 |
| 行为事件 | 冲突、违纪、欺凌、异常行为 | 识别安全和行为风险 |
| 家校沟通 | 电话、家访、面谈记录 | 识别家庭支持与协同情况 |
| AI 摘要 | 助手对话摘要、提醒结论 | 捕获高风险文本线索 |
| 关系信号 | 图谱共现、冲突关联、边缘关系 | 增强社交和安全判断 |

数据设计上遵循两个原则：

- 先用校园已有业务事实，尽量不依赖额外人工录入。
- 同一条记录既能服务展示，也能服务研判和复核。

## 4. 技术分层

### 4.1 基础规则层

基础规则层负责把结构化事实转为稳定、可控的基础风险分数。这里重点做四件事：

- 根据事件类型、频次和时间计算基础权重。
- 对近期信号给予更高权重，对历史信号做时间衰减。
- 对文本记录识别正向、负向和中性倾向。
- 识别保护性证据，避免系统只会累加负面信号。

除此之外，这一层还承担“边际递减”处理。也就是说，同类负向事件重复出现时，系统仍然会提高风险，但不会做无上限线性累加。这样可以避免单一事件类型把整张画像机械拉爆，更符合老师对“连续异常但强度递增趋缓”的实际判断。

这一层的目标是保证画像“站得住”，即使不调用智能体，也能形成一版可解释的学生画像。

### 4.1.1 安全披露捕获

系统并不只依赖结构化事件表，还会从 AI 对话摘要、教师备注和观察类文本中识别安全披露线索，例如被打、被威胁、害怕回班、持续恐惧等表达。

这类能力的作用是把文本里的高危苗头提前送入主链路，使系统在正式违纪事件形成之前，就能捕获更早的风险征兆。它既会影响基础画像，也会作为后续贝叶斯修正和智能体研判的重要输入。

### 4.1.2 保护性证据与误报消解

系统不是只会处理负向证据，还会显式处理保护性证据，例如：

- 同伴支持和共同活动
- 情绪或社交改善的观察记录
- 家校沟通中出现支持资源
- 教师复核确认“已缓解”或“误报”

这些信息会降低相关维度的后验概率，或直接削弱专项场景风险。这样做有两个目的：

- 让画像能真实反映“风险回落”而不是只升不降。
- 让系统能消解误报，避免老师对系统逐渐失去信任。

### 4.2 贝叶斯修正层

贝叶斯层不负责从零发现全部风险，而是对重点维度进行概率修正。它更适合处理两类问题：

- 多条证据不断累积之后，风险概率如何更新。
- 某类场景下，次生风险是否已经进入值得关注的区间。

当前它主要服务于高敏感维度和社交孤立等需要因果链判断的场景。

这里实际分成两部分：

- 通用贝叶斯辅助层：负责对情绪、安全、家庭、社交等维度做轻量后验修正。
- 贝叶斯网络层：负责恶性事件传播、社交孤立等更强调路径推断的专项场景。

也就是说，系统不是“只要提到贝叶斯就是同一套逻辑”，而是把通用修正和专项网络拆开实现。

### 4.3 关系增强层

图谱层用于处理“单条记录看不出来，但放在关系网络里就有意义”的信号。比如：

- 某学生长期处于班级互动边缘。
- 某学生与多个冲突事件节点共现。
- 某学生既有负向关系，也缺少正向支持关系。

这类信号往往很难通过单表聚合看出来，但对社交与安全维度有明显价值。

### 4.4 孤立专项插件

社交孤立是一个高价值但也高误判风险的场景，因此系统没有把它完全埋在通用画像里，而是单独设计了专项插件。

这层的主要目标是：

- 把社交孤立从“社交维度分数偏低”升级为“可专项解释的场景分析”。
- 让图谱信号、教师观察、复核状态和保护性证据能放在同一条分析链路里。
- 为前端提供单独的孤立分析入口，便于展示和专项调优。

目前它主要由 `backend/services/student_care_isolation_service.py` 承担，并通过 `backend/api/student_care.py` 暴露专项接口。

这层除了输出风险概率，还会输出数据覆盖度、缺失项提示、证据来源分组和专项置信度。也就是说，前端拿到的不只是“高不高”，还包括“为什么这样判断”和“当前判断是否足够有把握”。

### 4.5 多智能体解释层

多智能体层负责把“已经被整理过的事实与信号”解释成老师可理解的分析结果，而不是直接从原始数据库字段生成泛化结论。系统将任务拆成六个维度专家：

- 情绪专家
- 社交专家
- 安全专家
- 家庭专家
- 学习专家
- 行为专家

每个专家只对所属维度进行分析，最后由整合节点生成综合研判结论和建议。

## 5. 为什么采用多智能体

单一提示词虽然实现简单，但在学生关怀场景里存在三个明显问题：

- 一次性处理的信息太多，容易丢掉关键细节。
- 输出往往只有一段总评，不利于老师逐维理解。
- 某一维证据特别强时，会压制其他维度的判断。

多智能体方式的价值在于：

- 让不同维度分别聚焦自己的证据。
- 保留分维度结论，便于前端展示和教师复核。
- 让整合环节聚焦“总结和协调”，而不是兼顾全部细节。

在业务上，这种形式更贴近教师真实工作方式，因为老师本来就会同时从学习、家庭、社交、行为几个角度理解学生。

## 6. 教师复核与评估机制

学生关怀系统最终一定要回到教师手里，因此复核机制不是附属功能，而是核心设计的一部分。

系统当前通过复核与评估机制完成三件事：

1. 让老师确认系统判断是否可信。
2. 让系统知道哪些场景误报更高、哪些规则更有效。
3. 让后续调整有统计依据，而不是凭感觉修改权重。

复核评估页当前承担的价值主要有：

- 查看最近复核记录
- 观察系统命中和教师确认的一致情况
- 分析误报、真风险、处理中等状态分布
- 识别哪些规则或场景需要继续优化

这比单独放一张静态报表更有价值，因为它保留了“数据到动作”的闭环。

## 7. 主要前后端实现入口

下面这些文件是学生关怀体系的关键实现入口：

| 文件 | 主要职责 |
| --- | --- |
| `backend/services/student_care_service.py` | 关怀画像构建、基础维度评分、画像刷新 |
| `backend/services/student_care_agent_service.py` | 多智能体研判、综合结论、复核评估统计 |
| `backend/services/assistant_service.py` | 安全披露捕获与文本高危线索沉淀 |
| `backend/services/student_care_bayes_service.py` | 贝叶斯辅助层修正与证据更新 |
| `backend/services/student_care_isolation_service.py` | 社交孤立专项插件与贝叶斯网络分析 |
| `backend/services/student_care_graph_service.py` | 图谱关系增强与图信号生成 |
| `backend/services/student_care_bayes_config_service.py` | 贝叶斯规则的读取、覆盖和更新 |
| `backend/api/student_care.py` | 关怀能力对外接口 |
| `frontend/src/views/student/StudentList.vue` | 学生列表入口与画像触发入口 |
| `frontend/src/views/student/StudentCareDrawer.vue` | 单个学生画像和研判结果展示 |
| `frontend/src/views/student/StudentCareEvaluationPage.vue` | 研判评估、复核统计与系统概览 |

如果需要更明确地区分，可以这样理解：

- `student_care_bayes_service.py` 对应“通用贝叶斯辅助层”。
- `student_care_isolation_service.py` 对应“孤立专项插件里的贝叶斯网络层”。
- `assistant_service.py` 对应“安全披露捕获入口”。
- `student_care_service.py` 同时承接时间衰减、边际递减和保护性证据等基础规则处理。
- `student_care_graph_service.py` 对应“关系增强层的图谱同步、图信号和图视图输出”。
- `student_care_bayes_config_service.py` 对应“规则可运营配置能力”。

## 8.5 可运营配置与可视化能力

除了算法链路本身，当前实现还有两类常被忽略但很重要的能力：

1. 贝叶斯规则可配置化。
   系统支持通过接口查看和调整不同维度下的证据规则，而不是把所有先验和 LR 永久写死在代码里。
2. 图谱可视化与同步。
   系统支持查看图谱健康状态、同步单个学生子图、查看学生关系图，这让“关系增强”不只是后台黑盒，而是可演示、可校验的能力。

## 9. 页面职责划分

为了避免同一能力被多个页面重复表达，当前建议按下面方式理解页面分工：

| 页面 | 作用 |
| --- | --- |
| 学生列表 | 找到学生，进入个体工作流 |
| 关怀画像抽屉 | 查看单个学生的画像、风险和建议 |
| 研判评估页 | 观察系统表现、教师复核和整体趋势 |
| 标签相关页面 | 沉淀可复用的标签知识和审核结果 |

页面设计原则是：以教师动作链为主，而不是以技术模块拆页面。

## 10. 当前适合对外讲述的口径

如果面向答辩或汇报，建议把学生关怀模块描述为：

“一个把校园多源事实、规则评分、概率修正、关系增强和教师复核整合在一起的学生关怀智能研判体系。”

这样的表述比“多智能体风险研判系统”更稳，原因有三点：

- 它强调体系，而不是只强调模型结构。
- 它包含教师复核和评估闭环，更符合校园应用语境。
- 它不会把焦点过度压在单一技术名词上。

## 12. 一句话结论

学生关怀模块的重点不是“模型更复杂”，而是让复杂校园线索被稳定汇总、合理解释、可被教师复核，并最终形成持续优化的工作闭环。
